📌 PTBR과 페이지 테이블
PTBR (Page Table Base Register) — 현재 실행 중인 프로세스의 페이지 테이블이 메모리에 적재된 위치를 가리키는 특별한 레지스터.
  • 프로세스마다 고유 → PCB에 저장
  • 문맥 교환 시 해당 프로세스의 PTBR 값으로 변경됨 (그래야 새 프로세스의 페이지 테이블을 본다)
📌 TLB가 없을 때 메모리 접근 비용
한 번의 메모리 읽기 = 2회 접근
  1. 페이지 테이블 조회 (논리 주소 → 물리 주소 변환)
  2. 실제 프레임에 접근
→ 다단계 페이지 테이블이면 N+1회 (N단계 조회 + 1회 실제 데이터)

📌 TLB가 효과적인 원리
참조 지역성 (Locality of Reference) 덕분.
  • 시간적 지역성: 최근 접근한 주소를 곧 다시 접근할 가능성 ↑
  • 공간적 지역성: 한 번 접근한 주소 근처를 곧 접근할 가능성 ↑
→ 같은 페이지의 PTE가 TLB에 한 번 들어오면 그 페이지의 모든 접근에서 히트. 일반적인 프로그램은 TLB 히트율 95~99%.
⚠️ 책 표현 정정 #1: "TLB 히트 시 한 번만 메모리 접근"
(네가 노트에 이미 적어둔 정정) — TLB 히트 시 DRAM 접근은 0~1회. 데이터/명령 캐시(L1/L2/L3)에 히트하면 0회, 캐시 미스면 1회 이상. 즉 "TLB 히트" ≠ "DRAM 접근 1회" — 캐시 계층과 별개.
⚠️ 책 표현 정정 #2: "MMU가 페이지 워크 수행"
정확하지만 살짝 단순화. 두 모드가 있어:
  • Hardware-managed TLB (x86, ARM): MMU가 자동으로 페이지 워크
  • Software-managed TLB (MIPS, 일부 RISC): TLB 미스 시 OS의 트랩 핸들러가 직접 페이지 테이블 조회 → 더 느리지만 유연
🔍 깊이 보충 #1: 문맥 교환과 TLB
(Q1 답) 옛날에는 문맥 교환 시 TLB 전체 flush가 필요했어 — 다른 프로세스의 PTE를 그대로 쓰면 메모리 보호가 깨지니까. 하지만 매번 flush는 비용이 크다 (TLB가 다시 채워지는 동안 미스 폭증). → 해결책: ASID/PCID (Address Space Identifier / Process Context ID). 각 PTE에 프로세스 ID를 태그로 붙여서 TLB에 여러 프로세스의 PTE가 공존 가능. x86은 PCID, ARM은 ASID. 이 덕에 같은 프로세스로 돌아와도 TLB 미스가 적음.
🔍 깊이 보충 #2: TLB 미스 줄이는 전략
(Q2 답)
  • Huge Pages: 4KB 대신 2MB / 1GB 페이지 사용 — TLB 한 entry가 더 큰 영역을 cover. 데이터베이스/JVM에서 흔히 사용.
  • 접근 패턴 최적화: 큰 배열을 순차 접근하면 공간적 지역성 확보 (vs 랜덤 접근). 행 우선/열 우선 순서 일치시키기.
  • Working set 작게 유지: 한 번에 다루는 활성 메모리를 TLB가 cover 가능한 범위로
🔍 깊이 보충 #3: 멀티코어와 TLB shootdown
멀티코어 환경에서 한 코어가 페이지 테이블을 변경 (예: munmap)하면, 다른 코어의 TLB에 캐시된 stale entry를 무효화해야 해. 이를 TLB shootdown이라고 부르며, IPI(Inter-Processor Interrupt)로 다른 코어에 알림 → 각 코어가 자기 TLB의 해당 entry를 invalidate. 코어 수가 많아질수록 비용 ↑ → Linux는 mm_cpumask로 IPI 대상을 좁힘.
💡 한 줄 요약
EAT = h × (TLB시간 + 메모리시간) + (1-h) × (TLB시간 + 페이지워크 + 메모리시간)
TLB는 "주소 변환"이라는 비용을 95~99% 확률로 거의 0으로 만드는, 가상 메모리의 사실상 필수 부품.
