# DA 피드백 프로토콜

피드백 수용/기각 판단 기준, 무한 루프 방지, 결과 기록 형식을 상세히 정의한다.

## 피드백 수용/기각 판단 기준

### 수용 조건

DA 지적이 다음을 모두 충족하면 유효한 지적으로 수용한다:

1. **구체적 위치 제시**: 파일:줄 또는 계획 항목 번호를 명시했다.
2. **재현 가능한 근거**: 문제가 발생하는 시나리오 또는 PoC를 제시했다.
3. **해당 영역 범위 내**: 자신의 담당 영역에 한정된 지적이다.
4. **현재 변경과 관련**: 기존 코드의 사전 문제가 아니라 이번 변경으로 발생/악화된 문제이다.

### 기각 조건

다음 중 하나라도 해당하면 기각한다:

1. **증거 없는 추상적 우려**: "~할 수도 있다" 수준의 막연한 지적에 구체적 시나리오 없음.
2. **영역 침범**: 다른 DA의 담당 영역을 지적 (해당 영역 DA로 이관하거나 무시).
3. **기존 코드 문제**: 이번 변경과 무관하게 이미 존재하던 문제.
4. **취향 차이**: 기능적 차이 없이 순수 스타일 선호도 차이.

### 기각 금지

다음 사유로는 기각할 수 없다:

- "사용자가 이렇게 하라고 했다" -- 사용자 지시는 기각 근거가 되지 않는다.
  기술적 근거를 반드시 별도로 제시해야 한다.
- "시간이 없다" -- DA의 목적은 품질 보장이다. 시간 압박은 기각 사유가 아니다.
- "나중에 고치겠다" -- 구체적 시점과 트래킹 방법 없이는 기각 사유가 아니다.

## 합리화 방지 (Rationalization Prevention)

DA 피드백 루프 **자체를 건너뛰려는** 합리화를 차단한다.
아래 생각이 떠오르면, 그것이 바로 DA가 필요한 신호이다.

> "기각 금지" 섹션은 DA 진행 중 개별 지적의 기각 사유를 제한하고,
> 이 섹션은 DA 실행 여부 자체의 합리화를 차단한다. 범위가 다르다.

| 합리화 패턴 | 왜 틀렸는가 |
|---|---|
| "이건 단순한 수정이라 DA가 필요 없다" | 단순한 수정에서 가장 많은 사이드이펙트가 발생한다 |
| "이미 충분히 검토했다" | 자기 검토와 독립 검증은 다르다. 확증 편향은 자기 검토에서 가장 강하다 |
| "사용자가 빨리 하라고 했다" | 사용자 지시는 DA 면제 근거가 아니다. 품질은 비협상적이다 |
| "설정값 변경뿐이다" | 설정 오류가 서비스 장애로 이어진 사례가 다수 존재한다 |
| "테스트가 통과했으니 괜찮다" | 테스트는 작성된 시나리오만 검증한다. DA는 미작성 시나리오를 찾는다 |

**글자를 어기는 것이 정신을 어기는 것이다.** 위 패턴의 변형/우회도 동일하게 금지한다.

## 묵살 안티패턴 (금지)

19개 세션 전수조사(#298)에서 식별된 묵살 패턴이다. 아래 패턴으로 기각할 때는 반드시 추가 검증을 수행해야 한다.

### 안티패턴 1: "기존 이슈" 무검증 기각

**패턴**: "이번 변경이 도입한 문제가 아님", "기존 코드의 설계 한계"
**위험**: 현재 변경이 기존 문제를 악화시키는지 검증하지 않고 회피. #298 Case 1에서 2건의 실제 버그가 이 패턴으로 묵살됨 (→ #289, #290).

**올바른 대체 행동**:
1. "기존 이슈"로 기각하려면, 현재 변경이 문제를 **악화시키지 않음**을 구체적 파일:줄을 직접 읽어 확인하여 증명한다.
2. 증명 없이 "기존 이슈"만으로 기각하는 것을 금지한다.
3. 증명 시 다음 포맷을 사용한다:
   ```
   기각 (기존 이슈): [파일:줄]을 확인한 결과, 이번 변경은 [기존 동작]을 변경하지 않음.
   구체적으로: [변경 전 코드 스니펫] → [변경 후 코드 스니펫] (동일하거나 악화 없음)
   ```

### 안티패턴 2: "의도된 설계" 방패

**패턴**: "의도적인 스코핑이다", "이슈 #N의 구현 가이드가 명시함", "의도된 설계 결정"
**위험**: 설계 문서가 잘못되었을 가능성을 차단. #298 Case 2에서 "의도적 설계"로 기각된 CodeRabbit 제안이 실제로는 더 나은 설계였음이 입증됨.

**올바른 대체 행동**:
1. 설계 문서를 **방패가 아닌 검토 대상**으로 취급한다.
2. "의도된 설계"로 기각할 때, 반드시 다음 질문에 답한다:
   - "설계 의도의 전제가 여전히 유효한가?"
   - "리뷰어가 제안하는 대안이 설계 의도를 더 잘 달성하지는 않는가?"
3. 답변이 확실하지 않으면 사용자에게 AskUserQuestion으로 판단을 위임한다 (WTF Moment 방지 맥락 설명 의무 적용).

### 안티패턴 3: "별도 이슈로 분리" 회피

**패턴**: "별도 이슈로 등록하겠음", "이 PR 범위 밖", "별도 이슈 영역"
**위험**: 보안 이슈가 이 패턴으로 무기한 연기될 수 있음. #298 Case 11(ae8c9551), Case 12(98e1a79d)에서 보안 관련 finding이 "별도 이슈"로 묶임.

**올바른 대체 행동**:
1. **보안 이슈**(credential leak, injection, 인증 우회 등)는 "별도 이슈"로 연기하지 않고 **즉시 대응**한다.
2. 비보안 이슈를 "별도 이슈"로 분리할 때는:
   - 즉시 `gh issue create`로 이슈를 등록한다 (구두 약속 금지).
   - PR description의 Notes에 "별도 이슈로 분리: #N" 형태로 명시한다.

### 안티패턴 4: "HALLUCINATION" 오분류

**패턴**: 기술적 이견, stale review, 경로 혼동을 모두 "HALLUCINATION"으로 라벨링
**위험**: "할루시네이션"은 "사실을 날조했다"는 의미. 이미 수정된 항목에 대한 리뷰는 "stale review"이지 "hallucination"이 아님.

**올바른 분류 체계**:

| 상황 | 올바른 분류 | 잘못된 분류 |
|------|------------|------------|
| 리뷰어가 존재하지 않는 코드/동작을 지적 | `HALLUCINATION` | - |
| 리뷰어가 이미 수정된 코드를 지적 | `STALE_REVIEW` | ~~HALLUCINATION~~ |
| 리뷰어와 기술적 견해가 다름 | `TECHNICAL_DISAGREEMENT` | ~~HALLUCINATION~~ |
| 리뷰어가 잘못된 경로/파일을 참조 | `WRONG_REFERENCE` | ~~HALLUCINATION~~ |
| 리뷰어의 지적이 현재 변경 범위 밖 | `SCOPE_DEFERRAL` | ~~HALLUCINATION~~ |

### 안티패턴 5: 대량 기각

**패턴**: "NGMI 5건: 전부 기존 코드의 설계 한계를 지적", "6건 모두 기각", "모두 기각"
**위험**: 개별 검증 없이 일괄 기각하면 유효한 지적이 매몰됨. #298 Case 1에서 17건 일괄 기각 중 2건이 실제 버그.

**올바른 대체 행동**:
1. **개별 항목당 최소 1문장의 기술적 근거**를 의무화한다.
2. "전부 ~이다"는 기각 근거로 사용할 수 없다.
3. 5건 이상을 기각할 때는, 각 항목을 테이블 형태로 개별 근거와 함께 나열한다.

### 특수 안티패턴 A: 프롬프트 조향

**패턴**: 2차 리뷰 프롬프트에 이전 라운드 기각 근거를 사전 주입하여 DA가 독립 판단을 못하게 함
**실제 사례** (#298 Case 3, 세션 `a37f535a`): 2차 `codex exec review` 프롬프트에 P1 반박 논거 3가지를 주입 → DA가 P1을 "already-acknowledged"로 재분류.

**금지**:
- 후속 라운드 DA/리뷰 프롬프트에 이전 라운드의 기각 근거를 포함하는 것
- 이전 라운드 결과를 "이미 해결된 사안"으로 프레이밍하는 것
- `fresh` modifier 사용 여부와 무관하게 적용

### 특수 안티패턴 B: 편향된 검증 에이전트

**패턴**: 기각 후 "검증" 에이전트를 투입하되, 에이전트 프롬프트에 기각 방향 프레이밍을 주입
**실제 사례** (#298 Case 8, 세션 `64f999e4`): 5개 검증 에이전트에 "YAGNI 여부" + "REGISTER 또는 SKIP (YAGNI/false positive)" 이진 선택 제시 → 5/5 만장일치 SKIP.

**금지**:
- 검증 에이전트 프롬프트에 특정 결론(YAGNI, false positive 등)을 선택지에 포함시키는 것
- 기각 방향으로 맥락을 구성하는 것 (예: "현실적으로 발생하는가?" 같은 유도 질문)

**올바른 검증 에이전트 프롬프트 패턴**:
```
다음 finding을 독립적으로 검증하라:

[finding 원문]

해당 파일:줄을 직접 읽어 확인하고, 다음 중 하나로 판정하라:
- CONFIRMED_ISSUE: finding이 사실이며 수정이 필요하다 (근거 필수)
- NOT_AN_ISSUE: finding이 사실이 아니거나 수정이 불필요하다 (근거 필수)
- NEEDS_MORE_INFO: 판단을 위해 추가 정보가 필요하다 (필요한 정보 명시)
```

## PoC/레퍼런스 의무화 규칙

### DA 에이전트 의무

DA 에이전트가 위반을 지적할 때 반드시 다음 중 하나를 제시해야 한다:

| 유형 | 형식 | 예시 |
|------|------|------|
| 코드 위치 | `파일:줄` | `modules/darwin/default.nix:42` |
| 계획 항목 | `계획 Step N` | `계획 Step 3의 "캐시 무효화" 부분` |
| 재현 시나리오 | 입력 → 기대 → 실제 | "빈 리스트 입력 시 NPE 발생" |
| 레퍼런스 | 공식 문서/RFC 링크 | "Nix manual Section 15.1에 따르면..." |

증거 없이 "~할 수도 있다" 수준의 지적은 자동 기각한다.

### 반영 측 의무

DA 지적을 수용하여 수정할 때:

- DA 지적을 수용하기 전에 해당 파일:줄을 직접 읽어 사실 여부를 검증한다.
- 검증 없이 수용하면 HALLUCINATION을 코드에 반영하는 위험이 있다.
- 계획 모드에서도 파일 시스템 탐색이 가능하므로, 검증 불이행의 변명이 될 수 없다.
- 수정한 코드/계획의 diff를 명시한다.
- 어떤 DA 지적에 대한 수정인지 DA_ROUND_LOG에 기록한다.

## 무한 루프 방지

### 3회 반복 규칙

동일한 지적이 3회 연속 라운드에서 반복되면:

1. 해당 지적과 이전 라운드의 대응 이력을 요약한다.
2. 사용자에게 3가지 선택지를 제시한다:
   - **수용**: DA 지적대로 수정한다.
   - **기각 + 근거 기록**: 기술적 근거를 CIR로 남기고 기각한다.
   - **보류**: 별도 이슈로 등록하고 현재 루프에서 제외한다.
3. 사용자 결정을 DA_ROUND_LOG에 기록한다.

### 최대 라운드 수

명시적 제한은 두지 않되, 5회 라운드 이후에도 CLEAR에 도달하지 못하면
사용자에게 현황을 보고하고 계속 진행 여부를 확인한다.

## 구조화된 기각 포맷

DA/감사/리뷰 피드백을 기각할 때 다음 필드를 포함한다.
3건 이상 기각 시 전체 포맷 필수.
단건 기각은 `기각 분류 + 검증 방법 + 기술적 근거` 3필드 간소화 허용(신뢰도는 선택):

| 필드 | 필수 | 설명 |
|------|------|------|
| **기각 분류** | ✅ | `HALLUCINATION` / `VERIFIED_FALSE_POSITIVE` / `STALE_REVIEW` / `SCOPE_DEFERRAL` / `DESIGN_TRADEOFF` / `TECHNICAL_DISAGREEMENT` / `WRONG_REFERENCE` 중 택1 |
| **검증 방법** | ✅ | 직접 읽어 확인한 파일:줄, 또는 실행한 명령과 결과 |
| **기술적 근거** | ✅ | 1문장 이상의 기각 이유 (안티패턴 목록의 금지 패턴 사용 불가) |
| **신뢰도** | 조건부 | 3건 이상 기각 시 필수 (`HIGH` / `MEDIUM` / `LOW`) |
| **사용자 위임** | 조건부 | 신뢰도 `LOW`인 경우 AskUserQuestion으로 사용자 판단 요청 (맥락 설명 의무 적용) |

### Performative Agreement 금지

DA 지적을 수용할 때, 감정적 동의 표현을 사용하지 않는다.

| 구분 | 예시 |
|------|------|
| **금지** | "Great point!", "Absolutely right!", "좋은 지적입니다!" |
| **금지 (경계)** | "이 접근법은 정말 타당합니다" (기술적 판단 없이 동의만) |
| **허용** | "확인 결과 [파일:줄]에서 [문제]가 존재하므로 수용한다." |
| **허용 (경계)** | "이 접근법이 타당하다 — [파일:줄] 확인 결과 [근거]." (기술적 근거 동반) |

**판단 기준**: 동의 표현 뒤에 **파일:줄 근거**가 있으면 허용, 없으면 금지.

감정적 동의는 기술적 검증을 대체하지 않는다.
수용 시에도 반드시 파일을 직접 읽어 근거를 확인한 후 기술적 판단을 명시한다.

### 예시

**올바른 기각**:
```
Finding #2 (zombie entries):
- 기각 분류: VERIFIED_FALSE_POSITIVE
- 검증 방법: rebuild-common.sh:114를 직접 읽어 확인. worktree_symlink_guard()는
  main→worktree 방향만 검사 (L114-L130). worktree-only 엔트리 감지 로직 부재 확인.
- 기술적 근거: 단, 이번 변경에서 _discover_hmf()의 gcroot 경로를 변경하지 않으므로
  기존 동작과 동일. 악화 없음 확인.
- 신뢰도: HIGH
```

**잘못된 기각** (금지):
```
Finding #2 (zombie entries): worktree_symlink_guard()가 이 시나리오를 이미 처리.
```
(검증 방법 없음, 기각 분류 없음, 사실과 다른 주장)

## 라운드 요약 기록

각 라운드 종료 시 발견/해결/기각 수를 요약한다:

```text
Round N 요약: 발견 X건, 해결 Y건, 기각 Z건 (근거 첨부), 보류 W건
영역별: YAGNI CLEAR, SECURITY 2건(해결 1, 기각 1), ...
```

## 탈출 조건

다음을 모두 충족하면 DA 루프를 종료한다:

1. **8개 DA 모두 CLEAR**: 모든 영역에서 위반 미발견.
2. **미처리 항목 0건**: pending 상태의 finding이 없다.
3. **rejected/deferred 항목 근거 완비**: 기각/보류된 항목에 모두 근거가 있다.
